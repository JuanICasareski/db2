📖Caso: FlowOps

FlowOps es una plataforma SaaS multi-tenant que permite a empresas configurar procesos operativos simples mediante flujos definidos como datos. El sistema debe permitir crear una definición de proceso, iniciar una instancia, avanzar entre pasos, registrar eventos y consultar el estado.

El producto se inspira en herramientas de automatización de flujos como n8n o Make, pero el alcance del TPO no exige construir una herramienta equivalente completa. Se evaluará que los alumnos puedan modelar, persistir y ejecutar de forma mínima un flujo configurable usando bases NoSQL.

Formulario recibido → Validar datos → Tarea humana → Registrar decisión → Notificar resultado → Fin

🎯Objetivo general

Diseñar e implementar un prototipo mínimo de una plataforma BPM NoSQL que permita definir flujos operativos configurables, ejecutar instancias y conservar trazabilidad, justificando el uso de al menos dos modelos NoSQL vistos en la materia.

Diseño

Explicar arquitectura, entidades, flujos, particionamiento y decisiones de persistencia.

Persistencia

Usar al menos dos modelos NoSQL enseñados durante la cursada.

Prototipo

Implementar una API simple y persistencia real sobre los motores seleccionados.

Justificación

Defender decisiones con CAP, consistencia, escalabilidad y patrones de acceso.

📌Alcance obligatorio

El TPO debe limitarse a los siguientes elementos obligatorios:

  • Diseño multi-tenant simple con campo tenant_id en los datos principales.
  • Definición de procesos como datos configurables, preferentemente en formato JSON.
  • Diferenciación entre definición de proceso e instancia de proceso.
  • Persistencia real en al menos dos modelos NoSQL distintos vistos en la cursada.
  • Registro de eventos de ejecución o auditoría.
  • API mínima para crear/consultar procesos, iniciar instancias y consultar eventos.
  • Docker o Docker Compose para levantar los motores utilizados.
  • Análisis CAP y consistencia a nivel conceptual, aplicado a los subsistemas elegidos.

⚙️Requerimientos funcionales mínimos

1. Gestión de tenants

El sistema debe permitir representar empresas clientes mediante un identificador tenant_id. No se exige un módulo completo de administración de usuarios.

{ "tenant_id": "empresa_acme", "name": "ACME S.A.", "status": "active", "theme": { "primary_color": "#003366", "logo_url": "https://example.com/logo.png" } }

2. Definición de procesos

Un proceso se define como un documento con nodos y conexiones. Deben existir, como mínimo, nodos de inicio, formulario, decisión, tarea humana y fin.

{ "tenant_id": "empresa_acme", "process_id": "alta_proveedor", "version": 1, "name": "Alta de proveedor", "nodes": [ { "id": "start", "type": "start" }, { "id": "form", "type": "form" }, { "id": "validacion", "type": "decision", "condition": "cuit != null" }, { "id": "aprobacion", "type": "task", "assigned_role": "compras" }, { "id": "end", "type": "end" } ], "edges": [ { "from": "start", "to": "form" }, { "from": "form", "to": "validacion" }, { "from": "validacion", "to": "aprobacion", "when": "true" }, { "from": "aprobacion", "to": "end" } ] }

3. Instancias de proceso

Una instancia representa una ejecución concreta de una definición de proceso. Debe conservar estado actual y datos mínimos.

{ "tenant_id": "empresa_acme", "instance_id": "inst_1001", "process_id": "alta_proveedor", "process_version": 1, "status": "waiting_task", "current_node": "aprobacion", "data": { "razon_social": "Proveedor Demo S.A.", "cuit": "30-12345678-9" }, "created_at": "2025-10-31T10:15:00Z", "updated_at": "2025-10-31T10:20:00Z" }

4. Eventos

Se debe registrar qué ocurrió en la instancia: inicio, avance de nodo, creación o finalización de tarea, error simple o fin del proceso.

🗄️Persistencia NoSQL permitida

El grupo debe elegir al menos dos modelos NoSQL entre los vistos en clase. No es obligatorio usar todos.

Regla de alcance

Para aprobar, alcanza con implementar dos tecnologías NoSQL. Por ejemplo: MongoDB + Redis, MongoDB + Cassandra o MongoDB + Neo4j. El uso de tres o más tecnologías suma valor, pero no debe ser requisito excluyente.

📋Modelos de datos mínimos

El Plan de Sistemas debe incluir modelos lógicos y físicos para los siguientes elementos:

  • Tenant: empresa cliente y configuración básica.
  • ProcessDefinition: definición del flujo, nodos, conexiones y versión.
  • ProcessInstance: ejecución concreta de un proceso.
  • Task: tarea humana pendiente o completada.
  • Event: evento de auditoría asociado a una instancia.

Debe justificarse la clave principal o de particionamiento. Como mínimo se espera analizar el uso de tenant_id, process_id, instance_id y created_at.

🌐API mínima

Se debe implementar una API simple. Puede desarrollarse en el lenguaje que el grupo prefiera, siempre que conecte realmente con las bases NoSQL elegidas.

MétodoEndpointFunción
POST/api/{tenant_id}/processesCrear una definición de proceso.
GET/api/{tenant_id}/processes/{process_id}Consultar una definición de proceso.
POST/api/{tenant_id}/processes/{process_id}/instancesIniciar una instancia.
GET/api/{tenant_id}/instances/{instance_id}Consultar estado actual de una instancia.
POST/api/{tenant_id}/tasks/{task_id}/completeCompletar una tarea humana.
GET/api/{tenant_id}/instances/{instance_id}/eventsConsultar eventos de auditoría.

No se exige autenticación empresarial completa. Si se implementa seguridad, alcanza con una API Key simple documentada.

🏗️Arquitectura esperada

La arquitectura puede ser modular, sin exigir microservicios reales. Se espera que los alumnos separen conceptualmente responsabilidades.

Frontend simple / Postman / Swagger | v API de FlowOps | +--> Módulo de procesos --> MongoDB +--> Módulo de instancias --> MongoDB o Redis +--> Módulo de tareas --> MongoDB o Redis +--> Módulo de eventos --> MongoDB, Cassandra o Redis +--> Configuración tenant --> MongoDB o Redis

Si el grupo decide implementar servicios separados, puede hacerlo, pero no será obligatorio.

⚖️Evaluación CAP y consistencia

El análisis CAP debe ser aplicado y razonado, no meramente teórico. Debe responder:

  • Qué datos requieren mayor consistencia y cuáles pueden aceptar consistencia eventual.
  • Qué ocurre si se duplica un evento.
  • Qué pasa si una tarea humana se intenta completar dos veces.
  • Cómo se evitarían estados inválidos en una instancia.
  • Cómo se particionarían los datos por tenant, instancia o fecha.
  • Qué consultas se optimizan con cada modelo elegido.

Nivel esperado

No se pide implementar un sistema distribuido tolerante a particiones reales. Se pide comprender y justificar las consecuencias de las decisiones de modelado y persistencia.

🧪Prototipo funcional mínimo

Debe incluir

  • Repositorio con código fuente.
  • docker-compose.yml con al menos dos motores NoSQL.
  • API ejecutable localmente.
  • Datos de prueba para un flujo operativo.
  • Demostración de creación de proceso, inicio de instancia, avance de tarea y consulta de eventos.
  • Documento breve de instrucciones de ejecución.

Se acepta como interfaz

Frontend simple, Swagger, Postman, curl o una página HTML básica. No se exige un editor visual BPM.

📦Entregables

  1. Plan de Sistemas: dominio, arquitectura, modelos, persistencia, CAP y decisiones técnicas.
  2. Diagrama de arquitectura: componentes, bases NoSQL y flujos de datos.
  3. Modelo de datos: entidades/documentos/tablas/claves/índices según corresponda.
  4. Prototipo: código fuente y Docker Compose.
  5. Pruebas básicas: evidencia de endpoints funcionando y datos persistidos.
  6. Defensa oral: explicación de decisiones, limitaciones y mejoras futuras.

Condición central de aprobación

No aprobará una solución que sea solo un CRUD aislado. El sistema debe demostrar, aunque sea de manera mínima, la idea de flujo operativo: definición del proceso, instancia en ejecución, avance de estado y registro de eventos.
